Introduction

This Terraform module lives in Azure DevOps and provides a standard naming convention for various Cloud resources. It works for Azure, AWS, OCI, VMWare, and GCP. The goal is to maintain consistency and improve clarity in resource naming across different clouds and applications. This module follows a structured approach to naming resources, enhancing readibility. Users can customize descriptors and default variables by passing appropriate values through input variables.

Supported Cloud Platforms and Resource Coverage

  • Azure: 147 Azure resources supported with comprehensive coverage across:
  • API Management (azurerm_api_management → APIM)
  • Azure Databricks (azurerm_databricks_workspace → DBW)
  • Machine Learning services (azurerm_machine_learning_workspace → MLW)
  • Virtual Desktop Infrastructure (azurerm_virtual_desktop_* → VD*)
  • Monitoring & Analytics (azurerm_monitor_*, azurerm_stream_analytics → AG, DCR, SAC)

  • All core Azure services (VMs, Storage, Networking, Databases, etc.)

  • AWS: Comprehensive coverage of AWS resource types including EC2, S3, RDS, VPC, IAM, and many more

  • Oracle Cloud Infrastructure (OCI): 38 OCI resources covering:

  • API Gateway (oci_apigateway_gateway → APIGW)
  • Container Engine for Kubernetes (oci_containerengine_cluster → OKE)
  • Database services (oci_database_, oci_mysql_ → ADB, DB, DBS, MYSQLDB)
  • Functions (oci_functions_application → FNAPP)
  • Monitoring (oci_monitoring_alarm → ALARM)
  • Object Storage (oci_objectstorage_bucket → BUCKET)

  • Core networking and compute services

  • Google Cloud Platform (GCP): Google Cloud resource support including Compute Engine, Storage, BigQuery, and networking

  • VMware vSphere: vSphere resource naming support for on-premises virtualized environments

Publishing a new version of the module

Generally

1) Clone the repository.

2) Make changes to the code, either bug fixes or add resources.

3) Update CHANGELOG.md to reflect the changes made for a new version.

4) Update the README.md, preferably with terraform-docs so that the added resources are documented properly.

5) Add any new tests if you added resources.

6) Push changes to your branch.

7) Open a pull-request.

8) Make sure the build succeeds and check the draft KB article in ServiceNow.

9) Post-review, merge changes to main branch.

10) Check that the variable latest_terraform_universal_naming in the Latest Terraform Modules variable group has been updated to the correct published version.

12) Update modules to the new version.

Publishing your new version for the pipeline workflow

By default, merging changes to main will run a pipeline and create a new release artifact at the patch level (0.0.x). This should match the CHANGELOG.md version. If not, you will need to manually run the pipeline, select Custom for Choose publish version method parameter and enter the current version under Specific version to publish.

Adding new resource types

Resource types are treated as a string map per cloud. Resource types for each cloud are contained in the appropriate vars_[cloudname].tf file. For example to add OCI resource types, edit vars_oci.tf. To keep resource types input consistent please use what the cloud's terraform provider calls the resource as the key value in the map. As an example, for a VM in GCP, the Google provider calls the resource a google_compute_instance. In Azure you would use either an azurerm_windows_virtual_machine or an azurerm_linux_virtual machine. Attempts were made to keep the values returned by the map consistent across clouds. For instance, virtual machines or compute instances are called 'VM' regardless of cloud. This is not 100% true in all cases, but should be followed as much as possible to maintain consistency.

Adding new resource types involves also creating a unit acceptance test to insure the name is created appropriately. Unit tests should be placed in the corresponding vars_[cloudname].tf.tftest.hcl file in the tests folder. For example, if you are adding a resource to vars_gcp.tf, you need to add a unit test to tests/vars_gcp.tf.tftest.hcl. An example of the format of the test is below

run "validate_google_compute_instance_group" {
  command = plan

  variables {
    resource_type_input = "google_compute_instance_group"
  }

  assert {
    condition     = var.cloud_input != "gcp" ? true : local.resource_name == "ig-ent-app123401-n-use04"
    error_message = "Resource name did not match expected output for " \
                    "resource_type_input 'google_compute_instance_group'"
  }
}

Note: Name the test validate_[resource_name], put the resource name as the resource_type_input under variables, put the value of the abbreviation as the first part of the local.resource_name="" value. The rest of the name should follow the pattern [resource]-[division]-[workload]-[environment]-[location]. The rest of the input values are built from terraform.tfvars file and the variables listed at the top of the vars_[cloudname].tf.tftest.hcl file.

Adding new location descriptors

Location descriptors are region abbreviations used in most resource names at the end of the name to identify where a resource is deployed. These are also treated as a string map per cloud. Location descriptors for each cloud are contained in the appropriate vars_[cloudname].tf file. For example, to change or update a VMWare Datacenter location, edit vars_vmware.tf. For the clouds, the key is typically the long region or location name and the value is the shortened region. For example, in AWS, the primary region we use is 'us-east-1' which is abbreviated 'use01'. Currently location changes do not require an acceptance test.

Usage Instructions for pipeline workflow

  1. In the pipeline-build.yml make sure you have a task like the following:
    - task: UniversalPackages@0
      displayName: 'Download latest naming module'
      inputs:
          command: download
          vstsFeed: 'terraform_modules'
          vstsFeedPackage: 'terraform_global_naming'
          vstsPackageVersion: "$(latest_global_naming)"
  1. This code provides an example of how to use this module when generating a name for a resource:
    module "app_rg" {
      source                    = "./terraform_global_naming"

      business_unit_input       = var.business_unit
      cloud_input               = "azure"
      environment_input         = var.environment
      location_descriptor_input = var.location
      naming_index              = "01"
      resource_type_input       = "azurerm_resource_group"
      resource_type_override    = ""
      workload                  = var.workload
    }
  1. Now, you can use the Outputs from the app_rg module when specifying the resource group:
    module "VM_01_vm" {
      source                                = "./terraform_custom_vm_img_module"
      appid                                 =  var.appid
      business_unit_input                   =  var.business_unit
      location_descriptor_input             =  var.location
      environment_input                     =  var.environment
      location                              =  module.app_pattern.rg_location
      division                              =  var.business_unit
      resource_group_name                   =  module.app_rg.resource_name     <----------
      storageAccntURI                       =  module.app_pattern.sa_blob_endpoint
      availabilitySetId                     =  azurerm_availability_set.application.id
      tags                                  =  module.app_tags.def_vm_tags
      network_interface_ids                 =  module.VM_01_nic.nic_id[0]
      naming_index                          =  var.VM_01_naming_index
      countInt                              =  var.VM_01_countInt
      virtualMachineSize                    =  var.VM_01_virtualMachineSize
      OSMachineType_input                   =  var.VM_01_OSMachineType_input
      os_version_input                      =  var.VM_01_os_version_input
    }
  1. This code provides an example of how to use and test this module when generating a name for a resource:
    module "vm_name_01" {
        source = "registry.brunswick.com/aceo/naming/universal"
        version = "0.3.4"

        business_unit_input       = var.business_unit_input
        cloud_input               = "aws"
        environment_input         = var.environment_input
        location_descriptor_input = var.location_descriptor_input
        naming_index              = "01"
        resource_type_input       = "aws_instance"
        resource_type_override    = ""
        workload                  = var.workload
    }
  1. Now, you can use the Outputs from the vm_name_01 module when defining an EC2 instance.
    resource "aws_instance" "test-ec2" {
        provider               =  aws.euw1
        ami                    = "ami-083fc43c8bb28bd77"
        instance_type          = "t3.micro"
        subnet_id              = "subnet-06635fe80d0617863"
        vpc_security_group_ids = ["sg-0f1c111bcffbe2fda"]
        key_name               = "temp-key"

        tags  = {
            Name  = module.vm_name_01.resource_name    <----------
        }
    }

  1. Customize Input Variables.

    Add the values of the input variables according to requirements.

    Example:

    variable "business_unit_input" {
        type        = string
        description = "String value of business unit passed from parent"
        default     = "ENT"
    }

    variable "location_descriptor_input" {
        type        = string
        description = "String value of the location descriptor passed from parent"
        default     = "us-east-1"
    }

    variable "workload" {
        type        = string
        description = "Freehand workload descriptor for resource names " \
                      "(typically the APPID)"
        default     = "app1234"
    }

    variable "environment_input" {
        type        = string
        description = "String value of environment passed from parent"
        default     = "non_production"
    }

This code will create EC2 instance with the structured and consistent name
generated by the naming module, in this case 'vmentapp1234n'.
  1. You can test the module by using a terraform.tfvars file. Run terraform console It will load the terraform.tfvars so you can test the variables and outputs. See example below:
    business_unit_input       = "ENT"
    cloud_input               = "aws"
    environment_input         = "production"
    location_descriptor_input = "us-east-1"
    naming_index              = "01"
    resource_type_input       = "aws_lb"
    resource_type_override    = "NLBE"
    workload                  = "app1234"
